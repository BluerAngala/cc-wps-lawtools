# 路径结构修正说明

**日期：** 2025-11-14  
**类型：** Bug 修复

---

## 🐛 问题描述

在实际运行中发现，WPS 会锁定 `jsaddons` 下的项目目录，使其变为只读状态。因此无法在项目目录下创建配置文件子目录，导致以下错误：

```
[ERROR] 创建目录失败
{
  "dirPath": "C:/Users/11071/AppData/Roaming/kingsoft/wps/jsaddons/wps_lawtools_1.0.0"
}
```

---

## ✅ 解决方案

将配置目录从项目子目录调整为**项目同级目录**，路径结构变更如下：

### 之前（错误）

```
jsaddons/
└── wps_lawtools_1.0.0/          # 项目根目录
    └── config/                   # ❌ 无法创建，目录被锁定
        ├── config.json
        ├── templates/
        └── cache/
```

### 之后（正确）

```
AppDataPath/                          # C:/Users/xxx/AppData/Roaming
├── wps_lawtools_1.0.0_config/        # 配置目录（可写，在 AppDataPath 根目录）
│   ├── config.json
│   ├── templates/                     # 用户模板
│   ├── cache/                         # AI 缓存
│   └── logs/                          # 日志文件
└── kingsoft/
    └── wps/
        └── jsaddons/
            └── wps_lawtools_1.0.0/   # 项目根目录（只读，由 WPS 管理）
                └── public/
                    └── templates/     # 内置模板（只读）
```

---

## 🔧 修改内容

### 1. PathManager.js

**修改 `getConfigDir()` 方法：**

```javascript
// 之前
getConfigDir() {
  const projectRoot = this.getProjectRoot()
  return projectRoot ? projectRoot + '/config' : null
}

// 之后
getConfigDir() {
  const appDataPath = this.getAppDataPath()
  if (!appDataPath) return null

  const normalizedPath = appDataPath.replace(/\\/g, '/')
  return normalizedPath.replace(/\/+$/, '') + '/' + this.projectDirName + '_config'
}
```

**新增方法：**

```javascript
/**
 * 获取项目内置模板的实际文件系统路径
 */
getBuiltInTemplatesRealPath() {
  const projectRoot = this.getProjectRoot()
  return projectRoot ? projectRoot + '/public/templates' : null
}
```

**修改 `ensureAllDirs()` 方法：**

```javascript
// 移除项目根目录的创建（因为它是只读的）
const dirs = [
  // { name: '项目根目录', path: this.getProjectRoot() },  // ❌ 移除
  { name: '配置目录', path: this.getConfigDir() },
  { name: '模板目录', path: this.getTemplatesDir() },
  { name: '缓存目录', path: this.getCacheDir() }
]
```

## 📁 新的目录结构详解

### 1. 项目根目录（只读）

**路径：** `{AppDataPath}/kingsoft/wps/jsaddons/wps_lawtools_1.0.0`

**特点：**
- 由 WPS 管理，运行时会被锁定
- 只读，无法创建子目录或文件
- 包含项目代码和内置资源

**内容：**
```
wps_lawtools_1.0.0/
├── index.html
├── public/
│   └── templates/          # 内置模板（只读）
│       ├── templates.json
│       └── *.docx
└── ...其他项目文件
```

### 2. 配置目录（可写）

**路径：** `{AppDataPath}/wps_lawtools_1.0.0_config`

**特点：**
- 在 AppDataPath 根目录下（避开 jsaddons 只读限制）
- 完全可写，可以创建子目录和文件
- 存储用户数据和配置

**示例路径：**
```
C:/Users/11071/AppData/Roaming/wps_lawtools_1.0.0_config/
```

**内容：**
```
wps_lawtools_1.0.0_config/
├── config.json              # 应用配置
├── templates/               # 用户保存的模板
│   ├── templates.json
│   └── *.docx
├── cache/                   # AI 缓存
│   └── *.json
└── logs/                    # 日志文件
    └── *.log
```

## 🔄 模板文件处理

### 内置模板（只读）

**位置：** `{ProjectRoot}/public/templates/`

**访问方式：**
- HTTP 请求：`/templates/xxx.docx`
- 文件系统：`pathManager.getBuiltInTemplatesRealPath() + '/xxx.docx'`

**特点：**
- 随项目打包
- 只读，不可修改
- 用于初始模板

### 用户模板（可写）

**位置：** `{ConfigDir}/templates/`

**访问方式：**
- 文件系统：`pathManager.getTemplatesDir() + '/xxx.docx'`

**特点：**
- 用户保存的模板
- 可读写
- 持久化存储

## 🎓 经验总结

### 1. WPS 插件目录特性

- WPS 会锁定 `jsaddons` 整个目录（包括子目录）
- 项目目录在运行时是只读的
- 需要在 AppDataPath 根目录创建可写目录

### 2. 目录结构设计原则

- **只读资源**：放在项目目录（jsaddons 下）
- **可写数据**：放在 AppDataPath 根目录
- **版本隔离**：目录名包含版本号

### 3. 路径命名规范

- 项目目录：`{AppDataPath}/kingsoft/wps/jsaddons/{projectName}_{version}`
- 配置目录：`{AppDataPath}/{projectName}_{version}_config`
- 清晰区分，易于管理

### 4. 关键发现

⚠️ **jsaddons 整个目录都是只读的**：不仅项目目录本身，连 jsaddons 目录下的任何新建目录都无法创建。因此配置目录必须放在 AppDataPath 根目录下。
