# 流式与非流式请求说明

## 问题背景

在合同风险扫描中遇到了以下问题：

### 问题现象
```
[流式] 响应接收完成，总长度: 4523 字符
JSON解析失败: SyntaxError: Unexpected token '#'
```

### 原因分析
1. **流式输出不支持 JSON 模式**：SiliconFlow API 的 `response_format: { type: 'json_object' }` 参数在流式请求（`stream: true`）时不生效
2. **返回格式不匹配**：流式请求返回的是 Markdown 格式文本，但代码期望解析 JSON

### 根本原因
流式请求和 JSON 模式是**互斥**的功能特性：
- 流式请求：实时返回文本片段，适合展示进度
- JSON 模式：返回结构化数据，需要完整响应后才能保证格式正确

## 解决方案

### 方案选择
**使用非流式请求**作为合同审查的默认方式，原因：
1. ✅ 合同审查需要结构化的 JSON 输出
2. ✅ 需要准确解析 `issues` 和 `risks` 数组
3. ✅ 非流式请求也会显示进度信息（通过 `onProgress` 回调）
4. ✅ 用户体验影响不大（整体响应时间相同）

### 代码修改

#### 1. reviewAIService.js
```javascript
async reviewClause(context, contractType, options = {}) {
  // 合同审查必须返回 JSON 格式，因此默认使用非流式
  const needJsonFormat = options.response_format !== false // 默认需要 JSON
  const useStream = options.stream === true && !needJsonFormat
  
  if (useStream) {
    // 流式请求（仅用于展示，不解析为 JSON）
    response = await streamChatCompletions({ ... })
  } else {
    // 非流式请求（默认，支持JSON模式）
    response = await nonStreamChatCompletions({
      options: {
        response_format: { type: 'json_object' }
      }
    })
  }
}
```

#### 2. ContractRiskScan.vue
```javascript
const options = {
  autoApply: false,
  useCustomRules: false,
  // 不设置 stream: true，默认使用非流式（JSON 模式）
  onProgress: (progress) => {
    // 仍然可以显示进度信息
    scanProgress.value = {
      current: progress.current || 0,
      total: progress.total || 0,
      stage: progress.stage || '正在审查...'
    }
    // 显示响应内容预览
    if (progress.content) {
      realtimeResponse.value = `[${progress.stage}]\n\n${progress.content.substring(0, 500)}...`
    }
  }
}
```

## 使用场景对比

### 流式请求适用场景
✅ **适合**：
- 文档摘要生成
- 长文本翻译
- 创意写作
- 用户聊天对话
- 不需要结构化输出的场景

❌ **不适合**：
- 需要 JSON 格式输出
- 需要精确的数据结构
- 需要后续解析的场景

### 非流式请求适用场景
✅ **适合**：
- 合同审查（需要 JSON）
- 合同类型识别（需要 JSON）
- 全局分析（需要 JSON）
- 任何需要结构化输出的场景

✅ **优势**：
- 保证 JSON 格式正确
- 代码逻辑简单
- 错误处理容易

## 实际表现

### 非流式请求的用户体验
即使使用非流式请求，用户体验也很好：

1. **进度显示**：
   ```
   [正在审查条款...]
   [审查完成]
   ```

2. **响应预览**：
   ```
   [审查完成]
   
   {"issues": [{"severity": "high", "comment": "..."}], "risks": [...]}
   ```

3. **总耗时**：与流式请求相同（都需要等待完整响应）

### 流式请求的价值
流式请求的真正价值在于**心理感知**，用户可以看到"AI 正在思考"，但对于：
- 需要 JSON 的场景：流式反而会导致解析错误
- 短时间响应（<5秒）：流式的优势不明显

## 最佳实践

### 推荐配置

#### 合同审查类功能（需要 JSON）
```javascript
{
  stream: false,  // 或不设置（默认）
  onProgress: (progress) => {
    // 显示阶段信息
    console.log(progress.stage)
  }
}
```

#### 文本生成类功能（展示为主）
```javascript
{
  stream: true,
  onChunk: (chunk) => {
    // 实时显示文本片段
    appendToUI(chunk)
  }
}
```

### 配置优先级

```javascript
// reviewClause 的逻辑
const needJsonFormat = options.response_format !== false  // 默认 true
const useStream = options.stream === true && !needJsonFormat

// 结果：
// 1. 默认情况（无配置）：非流式 + JSON
// 2. stream: true：仍然使用非流式（因为需要 JSON）
// 3. stream: true + response_format: false：使用流式
```

## 性能对比

### 非流式请求
```
开始请求 → 等待完整响应 → 一次性返回
总耗时: ~3-10秒
```

### 流式请求
```
开始请求 → 逐块返回 → 完成
总耗时: ~3-10秒（相同）
优势: 更早看到部分内容（心理感知更好）
劣势: 不支持 JSON 模式
```

## 总结

### 当前实现
- ✅ 合同审查：默认使用**非流式 + JSON 模式**
- ✅ 合同类型识别：使用**非流式 + JSON 模式**
- ✅ 全局分析：使用**非流式 + JSON 模式**
- ✅ 保留流式能力：未来可用于其他场景

### 用户体验
- ✅ 仍然有进度提示
- ✅ 可以看到响应内容预览
- ✅ 结构化数据解析稳定
- ✅ 不会出现 JSON 解析错误

### 技术决策
**非流式请求是合同审查的正确选择**，因为：
1. 保证数据格式正确
2. 代码逻辑清晰
3. 错误率更低
4. 用户体验不受影响（有进度显示）

流式请求保留用于未来的**纯文本生成**场景，如文档摘要、润色等。
